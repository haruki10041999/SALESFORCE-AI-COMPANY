# フロー設計

## 概要
Salesforce Flow の設計品質を保ち、過剰複雑化と保守困難化を防ぐ指針。

## いつ使うか
- 新規自動化要件の方式選定時
- 既存 Flow の複雑化が問題になったとき
- Flow と Apex の責務分担を決めるとき

## 重要な原則
- Flow は単純な自動化に使い、複雑なロジックは Apex に委譲する
- 1 Flow に詰め込まず、サブフローで分割する
- デプロイ前に必ず非アクティブ版を保存する

## プラットフォーム固有の制約・数値
| 項目 | 上限 |
|---|---|
| Flow インタビュー数（同時実行） | 2,000 / org |
| 1 Flow 内の要素数（推奨上限） | 50以下（保守性確保のため） |
| Record-Triggered Flow の DML | Apex と共有。SOQL/DML limit に注意 |

## フローとApexの選択基準
| 条件 | 推奨 |
|---|---|
| 条件分岐が 5 以下 | Flow |
| 条件分岐が 10 以上 | Apex |
| 外部 Callout が必要 | Apex |
| 複数オブジェクトをまたぐ複雑な集計 | Apex |
| 画面入力フォームが必要 | Screen Flow |
| 管理者が設定変更できる必要がある | Flow |

## よい例・悪い例
### 悪い例
- 1つの Flow に 30 以上の分岐と 5 種のオブジェクト更新を詰め込む

### よい例
- 判定ロジックを InvocableMethod の Apex に切り出し、Flow はオーケストレーションのみ担当

## チェックリスト
- Flow の要素数が 50 以下か
- Apex Trigger との実行順序を確認したか
- バージョン管理と非アクティブ版の保存をしたか
- 管理者が内容を理解・変更できるシンプルさか
